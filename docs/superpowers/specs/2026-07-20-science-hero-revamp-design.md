# 科學英雄改版設計規格 — 全套八角理論強化計畫

## 背景與目標

孩子覺得科學英雄不如字鬥英雄好玩。本 session 已先補上第一波（PvE 對戰、同裝置
PvP、科學夥伴助戰、段位排行），但長線黏著層仍然空白。字字珠璣 2026-07-20 已完成
同性質的四子系統改版計畫（設計規格＋四份 TDD 計畫，38 Tasks），本規格是科學英雄
的對等版本：**即時對戰、精靈融合、市場交易、科學基地**四大子系統一次排定，用
Octalysis 八角理論逐項檢查，主要驅動力盡量拉到 9 分——並針對科學英雄自身現況
因地制宜，不照抄字字珠璣換皮。

## 與字字珠璣的三個關鍵差異（四份計畫都必須遵守）

1. **科學英雄沒有既有的 meta 層**：無貨幣、無收藏圖鑑、無裝備、無班級碼、無
   `js/meta/`。四份計畫要自帶地基——晶能經濟由「科學基地」計畫建立，班級碼由
   「即時對戰」計畫（全班戰況牆 Task）引入，市集消費這兩者。
2. **模組慣例是 `<script>` IIFE 全域掛載，不是 ES module**。新模組一律
   `const SciXxx = (() => { ... return {...}; })();` 放 `js/` 平層（不開 `js/meta/`），
   `index.html` 依相依序加 `<script>` 標籤。單元測試沿用 `test/logic.test.mjs`
   的 vm 串接 harness（把檔案串成一支 script 跑進 `vm.createContext`，用
   `globalThis.__exports = {...}` 取出），煙霧測試沿用 `test/smoke.mjs` 的
   playwright-core 模式。計畫裡的介面規格與測試範例必須用這套慣例。
3. **視覺目前以 emoji 為主**（對手/夥伴皆 emoji）。生圖清單照排但**開發不阻塞**：
   UI 一律先用 emoji/佔位圖上線，實體美術另批 codex exec 生圖後換裝。

## 科學英雄現況介面（實作計畫引用的既有事實，寫計畫前已核實）

- 資料：四科 1001 詞條（`data/elementary.json` 251、biology 250、physics-chemistry
  250、earth-science 250），`SUBJECTS` key＝`nature`/`biology`/`chemphys`/`earth`。
- 存檔：`SciStore`（`js/store.js`），key `science-hero:v1`，state＝
  `{ cards:{id:{box,due,seen,wrong}}, stats:{streakDays,lastActiveDate,totalReviews}, rank:{pts,peak,shieldWk}, battle:{beaten:[]}, weakLog:[...] }`。
  API：`load/save/getCard/setCard/touchDailyStreak/todayStr/bumpDailyCount`。
- 答題唯一記錄出口：`app.js` 的 `recordAnswer(target, correct, elapsedMs)`（內呼
  `SciWeak.recordAnswer`＋`SciFlashcard.bumpBox`＋`totalReviews+=1`＋save＋音效）。
  **所有新系統的「答對/答錯」鉤子都從這裡接，不得另闢記錄路徑**。
- 出題：`SciQuiz.buildQuestion(target, pool)` 四選一；`currentPool()` 是分頁/單元/
  年級篩選後的唯一濾字點。
- 對戰：`SciBattle`（`js/battle.js`）匯出 `OPPONENTS/TIER_UNLOCK/isUnlocked/
  calcDamage/mount/RANKS/rankInfo/rankWin/rankLose/weekStr/COMPANION_TIERS/companionFor`。
  傷害 `calcDamage(combo, hp)=12+combo*3`（hp<30 ×1.5）；PvE 段位勝+20/敗-10、
  每週首敗保護；夥伴 `companionFor(masteredCount)` 五階（🥚→✨）純函式推導。
- 精通：`masteredCardCount()`（app.js）＝box 升滿的卡數；精通稱號 `RANK_TIERS`
  五階（見習生 0→初階英雄 1→進階英雄 10→資深英雄 30→領域專家 80）。
- 測試：`node scripts/validate-all.mjs`、`node --test test/logic.test.mjs`（13 綠）、
  `node test/smoke.mjs`（playwright，11 檢查綠）。
- 部署：三平台（Vercel＋alias `science-hero-hk6429.vercel.app`／CF Pages
  `science-hero.pages.dev`／Netlify），母版 `naicheng-claude-agent/科學英雄`、
  部署副本 `~/projects/science-hero`（branch `master`）。

## 技術架構總覽

**統一後端**：新增 Cloudflare Pages Functions + D1（比照字字珠璣/字鬥英雄已驗證
架構）。**後端只集中 Cloudflare Pages 一個平台**；其餘鏡像站透過統一的
`SHAPI.call()` helper 打絕對網址 `https://science-hero.pages.dev/api/...`——修正
vocab-duel `fetch('api/room')` 相對路徑導致鏡像站後端降級的地雷，第一天就做對。

**跨子系統契約（所有計畫不得漂移）**：
- `SHAPI.call(path, body)`（`js/shapi.js`，IIFE）＝前端唯一後端入口；離線/失敗回
  `{ok:0, error}` soft error，前端一律優雅降級。
- D1 資料表命名空間前綴：即時對戰 `rt:`、市集 `mkt:`（**融合純前端，不碰後端**）。
- HMAC secret 分離：`RT_SECRET` 與 `MKT_SECRET` 各自獨立，單點外洩不波及他系統。
- 後端 kv 層：`functions/lib/_kv.js` D1 shim（Redis 風格 get/set/hset/zset 指令，
  照抄字字珠璣 rtbattle 計畫 Task 2/3 的 schema＋假 D1 測試 helper 手法）。
- 新貨幣「**晶能**」：收支只走 `SciEconomy`（`js/economy.js`）的
  `earnCrystals / spendCrystals` 單一入口；每日獲取上限 100（一次性成就獎勵不計）；
  數值一律掛真實學習量（答對/精通/勝場），不掛操作次數。
- 新 localStorage 鑰匙各系統一把：`sci_econ`（晶能）、`sci_base`（基地佈局）、
  `sci_fusion`（精靈/稚靈）、`sci_class`（班級碼＋暱稱，rtbattle 建立、market 共用）、
  `sci_market`（本機留痕/限購快取）。全部讀寫包 try/catch。

**與現有引擎的介面**：`SciQuiz`/`SciBattle.calcDamage` 不改。即時對戰＝雙方本機用
同一亂數種子從同一 pool 出同一串題，伺服器只當「狀態郵筒」轉發傷害與心跳（字鬥
英雄已驗證模式）；融合稚靈助戰走既有 companion assist 通道（`mount` 內加傷/回血
的同一路徑）擴充，不另闢戰鬥運算。

## 四大子系統設計摘要

### 1. 科學基地（base，對應字靈書院）

- 「科學研究基地」自由式佈局：中央**基地主樓五階**直接綁精通稱號 `RANK_TIERS`
  （見習營帳→初階研究站→進階實驗樓→資深研究院→領域總部）；四周**四科展館**
  （自然園圃／生物標本館／理化實驗室／地科天文台）各自讀該科精通%換繁茂度
  （荒蕪/初萌/漸盛/繁茂/鼎盛，門檻 0/10/30/60/100 對齊單元里程碑）。
- **1001 張詞卡實體化陳列**：精通卡依科別換算成展館裝飾（自然=植栽昆蟲箱、生物=
  標本罐顯微鏡、理化=燒杯儀器、地科=礦石星象儀），品階依煉成過程（零錯=金級、
  錯 1 次=銀級、其餘=銅級）。看得到自己「養」出來的基地。
- **賦予創造力主攻**（字鬥/字字兩站最弱項）：自由拖曳擺放（座標存檔）、裝飾樣式
  可用晶能換購、門牌/研究銘言從預設詞庫命名（不開放自由輸入）。
- **晶能經濟在此計畫建立**（`SciEconomy`＋`recordAnswer`/對戰結算掉落鉤子），供
  融合與市集共用。
- 慶典佇列＋成就牆（段位/連對/守繼紀錄陳列）；生圖清單另批。

### 2. 即時對戰（rtbattle）

- **配對**：4 位數房號＋10 分鐘 TTL（同步）；6 碼挑戰書＋7 天 TTL（非同步，不需
  雙方同時在線）。1.5 秒輪詢、非 WebSocket；伺服器只存進度/心跳，不存題目。
- **同 seed 出題**：雙方本機用同一 seed 對同一科目 pool 洗題，全程同題同選項序。
- **科學奇遇**（seed 決定事件序列，效果只影響觸發方，全部正面白帽）：靈感閃現
  （下題傷害×2）／實驗突破（排除一個錯誤選項）／能量湧現（回 10 HP）／護目鏡
  （下一次答錯不斷連擊）。科學英雄目前沒有奇遇層，本計畫內建輕量版。
- **全班戰況牆**：老師開房、全班同題搶答；**榜只露前五**＋「你目前第 N 名」私訊
  自己，不公開倒數名單（白帽）。此 Task 順帶引入班級碼（`sci_class`）。
- **月賽季**：段位分上雲（D1 zset）、每月重置、賽季稱號發放。

### 3. 精靈融合（fusion，對應神獸融合）

- **先把單一夥伴擴展為四科精靈**：每科一隻（各綁該科精通卡數，沿用
  `companionFor` 純函式推導手法），出戰時用當前科目的精靈助戰——數值仍然
  100% 掛真實學習量。
- **融合**：兩科精靈皆滿階（該科精通 ≥100）＋雙科近期正確率 ≥80% 才解鎖；
  四科兩兩配對恰好 **6 隻稚靈**封頂（自然×生物=森靈鹿、自然×理化=晶石狐、
  自然×地科=風嵐鷹、生物×理化=煉金龍、生物×地科=深海鯨、理化×地科=星核獸）。
- **雙親不消耗**（硬規則，沿用字鬥英雄 P3-11 教訓）；失敗只扣晶能＋安慰台詞＋
  少量返還，精靈與學習進度完全不受影響（白帽：代價走時間成本）。
- **配方揭曉解謎**：下一隻稚靈的長相/配方預設剪影，答對該兩科的隱藏題才揭曉。
- 稚靈可隨行出戰（走既有 assist 通道疊加）、自訂暱稱、**稚靈名片**canvas 分享卡。
- **純前端**，不碰後端。

### 4. 科學市集（market，對應翰墨集市）

- **交易物限定「實驗道具」與「基地裝飾部件」，精靈/稚靈不可交易**（夥伴不是商品）。
- **班級限定掛單**（吃 rtbattle 建立的 `sci_class`），全站公開另設開關預設關。
- 伺服器端**驗貨＋HMAC 簽章＋限流**；價格帶依品階公開設定，無隱藏演算法。
- **每日限購 3 件**；**週五限時開市**（稀缺性）；賣家收款走 claimKey＋10% 稅。
- **感謝小卡全走預設選項**，無自由文字輸入。
- 風險五項逐一對應：金錢化聯想（規則頁明寫晶能不可兌現）／社交壓力（無自由文字、
  榜單白帽）／跨班風險（預設班級限定）／公平性（價格帶公開）／共用後端資安
  （`mkt:` namespace＋`MKT_SECRET` 分離）。

## 八角理論總表

| 驅動力 | 現況初評 | 主要負責子系統 | 拉到 9 分的關鍵補強 |
|---|---|---|---|
| 1. 意義使命 | 2 | 科學基地（→8） | 「重建科學研究基地」敘事：精通=把失落的知識標本迎回展館，主樓升階觸發里程碑劇情卡 |
| 2. 成就感 | 5（段位已上線） | 即時對戰（賽季）、基地成就牆 | 賽季稱號＋可分享戰報卡；基地成就牆陳列連對/守繼/段位巔峰紀錄 |
| 3. 賦予創造力 | 1 | **科學基地（1→9，全站最弱項也是最大機會）** | 自由拖曳擺放＋裝飾樣式換購＋門牌銘言詞庫命名；融合的被動二選一 |
| 4. 擁有感 | 4（夥伴已有） | 融合（→9）、基地（→9）、市集 | 四科精靈＋稚靈暱稱＋名片；「我蓋的基地」；市集交易物連結基地裝飾 |
| 5. 社交影響 | 2（同裝置 PvP 已有） | **即時對戰（2→9）**、市集 | 房號跨裝置對戰＋挑戰書＋全班戰況牆＋分享卡；市集感謝小卡＋集市達人 |
| 6. 稀缺性 | 3 | 即時對戰、融合、市集 | 賽季制；稚靈剪影渴望牆＋封頂 6 隻；週五開市＋每日限購＋珍品週限量 |
| 7. 未知性 | 2 | **即時對戰奇遇（2→8）**、融合（→9） | seed 化科學奇遇事件；配方揭曉解謎（答對隱藏題才見下一隻稚靈） |
| 8. 損失規避 | 4（首敗保護已有） | 融合、市集 | 失敗只扣晶能走時間成本；蒙塵類懲罰一律不做；市集無懲罰性設計 |

**設計原則**：損失規避一律選時間成本而非資源扣減/懲罰；全部符合 vocab-game-pipeline
skill 鐵律（養成數值掛真實學習量、戰鬥是 quiz 引擎薄包裝、PvP 用 seeded 確定性題目）。

## 風險與注意事項（國小/國中生使用情境）

1. 晶能不可兌換現實金錢或禮物（規則頁明寫）；無任何真錢加值管道。
2. 市集/小卡/命名全走預設選項，無自由文字輸入，杜絕霸凌管道。
3. 戰況牆只露前五＋私訊自己名次，不公開倒數名單。
4. 市集預設班級限定；暱稱驗證與限流在伺服器端。
5. `rt:`/`mkt:` namespace 與 `RT_SECRET`/`MKT_SECRET` 分離。

## 建議實作優先順序

1. **科學基地基礎迴圈**（晶能經濟＋主樓/展館＋詞卡陳列）——不依賴後端、地基最厚
2. **即時對戰核心**（SHAPI＋D1 shim＋房號輪詢）——技術路徑已被兩個姊妹站驗證
3. **即時對戰奇遇＋全班戰況牆**（引入班級碼，市集的前置）
4. **精靈融合**（四科精靈擴展→融合核心→解謎/名片）
5. **科學市集**——風險評估最複雜、依賴後端防作弊最成熟，放最後
6. 各系統八角補強細節（分享卡/賽季/慶典）穿插在對應子系統完成後補上

## 美術資產排產（開發不阻塞，另批 codex exec 生圖）

- 基地場景元素：主樓五階＋四展館各五級繁茂度＋裝飾多品階（估 ~30 張）
- 稚靈立繪 6 張＋四科精靈終階 4 張（估 10 張）
- 即時對戰勝負畫面／市集主視覺（估 4–6 張）
- 風格待定案後統一（現站為 emoji，先上 emoji/佔位、完工後換裝）

## 範圍界線（本次不做）

- 不做 WebSocket（輪詢已足夠）；不做跨裝置即時 PvP 以外的帳號系統
- 融合不做排列組合全融合樹（收斂為四科兩兩=6 隻封頂）
- 市集不開放自由文字聊天/喊價；精靈不可交易
- 不做懲罰性損失規避設計（教育產品白帽原則）
- 不動 `SciQuiz`/`SciFlashcard`/`SciWeak` 既有函式簽名
